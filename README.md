# DS 4320 Project 2: Predicting 30-Day Hospital Readmission Risk Using MongoDB

### Executive Summary
This repository contains a complete end-to-end data science pipeline focused on predicting short-term hospital readmission risk using a MongoDB document database. Synthetic electronic health record data generated by Synthea is acquired, transformed into a patient-centered document model, and stored in MongoDB Atlas. The project then queries the database into an encounter-level analysis dataset, engineers clinically relevant features, and applies supervised machine learning models to estimate whether a patient will be readmitted within 30 days of discharge. Final outputs include publication-quality visualizations, a press release summarizing findings, and fully documented metadata describing the dataset, methods, and decisions used throughout the project.

## Problem Definition

**General Problem**    
Predicting hospital readmission risk

**Refined Problem**    
Using patient-level clinical and demographic data stored in a MongoDB database, predict the likelihood of a patient being readmitted within 30 days of discharge and identify key risk factors associated with readmission.

**Rationale for refinement**   
The refined problem focuses specifically on 30-day readmission prediction because it is a widely used benchmark in healthcare policy and quality assessment. Narrowing the scope makes the project more actionable and measurable while still addressing the broader issue. Additionally, structuring the data in MongoDB allows for flexible storage of complex patient records, including nested information like conditions, medications, encounters, and procedures. This makes it easier to integrate multiple aspects of a patient’s history into a single model, which would be more cumbersome in a strictly relational schema. The refinement therefore aligns both with real-world relevance and with the strengths of a document-based database.

**Motivation**  
Hospital readmissions are both costly and, in many cases, preventable. Healthcare systems are increasingly evaluated based on readmission rates, and reducing these rates can improve patient outcomes while lowering costs. With the growing availability of electronic health records (EHRs), there is an opportunity to leverage data to identify high-risk patients before they leave the hospital. By building a predictive system, hospitals can allocate resources more effectively—such as scheduling follow-ups, improving discharge instructions, or prioritizing care coordination for at-risk individuals.

## Data Creation

### Provenance

The raw data for this project was obtained from Synthea, an open-source synthetic healthcare data generator developed to simulate realistic electronic health records while avoiding the privacy constraints associated with real patient data. Synthea produces longitudinal patient histories that include demographics, encounters, diagnoses, medications, procedures, observations, providers, and organizational records. For this project, the publicly available CSV export archive was downloaded and processed using automated Python scripts.

After download, only the subset of CSV files directly relevant to predicting hospital readmission risk was retained. These included patient demographics (`patients.csv`), encounter histories (`encounters.csv`), diagnoses (`conditions.csv`), medications (`medications.csv`), procedures (`procedures.csv`), clinical observations (`observations.csv`), and provider/organization reference tables. The selected files were then transformed into a MongoDB document model where each patient is stored as a single document with embedded encounter-level clinical history. This patient-centered structure supports efficient querying for predictive analytics and better reflects how healthcare records are commonly reviewed in practice.

### Bias Identification

Although Synthea avoids direct privacy concerns, synthetic data can still contain bias. The generative rules used to simulate patients, diseases, treatments, and healthcare access may not fully capture real-world disparities across race, age, geography, insurance status, or provider quality. In addition, healthcare utilization patterns in synthetic data may be more regularized than those seen in real populations, which can affect readmission modeling.

### Bias Mitigation

Bias was addressed by explicitly including demographic variables as measurable predictors rather than ignoring them, allowing performance to be compared across groups if desired. Multiple model types were evaluated to reduce dependence on one algorithmic assumption. Because the data is synthetic, findings are interpreted as methodological rather than clinical claims. Future extensions could quantify fairness metrics across subgroups, recalibrate predictions by population segment, or validate the modeling pipeline on real de-identified hospital data.

### Rationale for Critical Decisions

Several design decisions were made to align the dataset with the refined project objective. First, only clinically relevant CSV files were selected rather than importing every available Synthea table. This reduced unnecessary complexity while preserving the variables most likely to influence readmission risk. Second, a document-oriented MongoDB schema was chosen instead of a relational schema because patient histories are hierarchical and longitudinal. Embedding encounters and related clinical events within patient documents makes patient-level analysis more natural and reduces repeated joins.

The target variable for readmission was operationalized as whether a patient experienced another encounter within 30 days after discharge. This required a judgment call because synthetic datasets do not always encode policy-ready readmission labels directly. As a result, the project uses a consistent rule-based approximation. Additional uncertainty may arise from missing timestamps, encounter type differences, or synthetic generation assumptions. These issues were mitigated through data cleaning, timestamp validation, removal of invalid lengths of stay, and comparison of multiple predictive models rather than relying on a single method.

## Metadata

### Implicit Schema

The MongoDB database uses a patient-centered document model in which each document represents one unique patient. Top-level fields store demographic and identifying attributes, while clinical history is embedded as nested arrays of encounter records. Each encounter may contain additional nested arrays for diagnoses, medications, procedures, and observations associated with that visit.

This schema was selected because healthcare records are naturally hierarchical and longitudinal. Rather than spreading patient history across many relational tables, the document model keeps clinically related information together. This supports efficient patient-level queries, simplifies feature engineering, and better reflects how healthcare data is often reviewed in practice.

**General schema outline:**

```json
{
  "_id": "patient_id",
  "BIRTHDATE": "...",
  "GENDER": "...",
  "RACE": "...",
  "ETHNICITY": "...",
  "encounters": [
    {
      "Id": "...",
      "START": "...",
      "STOP": "...",
      "ENCOUNTERCLASS": "...",
      "readmitted_30_days": true,
      "conditions": [],
      "medications": [],
      "procedures": [],
      "observations": []
    }
  ]
}
```
